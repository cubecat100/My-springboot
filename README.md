# PaintChat

> 실시간 그림 채팅방을 구현한 Spring Boot 포트폴리오 프로젝트

## 링크

- 배포 : [배포 링크](https://port-0-my-springboot-mmnystl5eb755ba2.sel3.cloudtype.app/)

<!-- 시연 영상 예시: [영상 링크](https://www.youtube.com/watch?v=your-demo) -->

실시간 그림 채팅방을 구현한 Spring Boot 포트폴리오 프로젝트입니다.  
사용자는 게스트 닉네임으로 바로 입장할 수 있고, 방에 들어오면 그동안 저장된 그림 이력이 순차적으로 재생됩니다. 재생이 끝난 뒤에는 같은 캔버스 위에서 다른 사용자와 함께 그림을 이어서 그릴 수 있습니다.

이 프로젝트는 단순한 그림판이 아니라, 다음 두 가지를 함께 해결하는 데 초점을 맞췄습니다.

- 여러 사용자가 동시에 그리는 실시간 협업 경험
- 늦게 입장한 사용자도 문맥을 이해할 수 있는 그림 이력 복원 경험

## 프로젝트 개요

### 한 줄 소개

WebSocket 기반 실시간 드로잉, DB 기반 그림 이력 저장, 입장 시 재생 기능을 결합한 멀티 유저 그림 채팅방 서비스입니다.

### 개발 목표

- 로그인 없이 바로 체험 가능한 진입 구조 만들기
- 실시간 드로잉과 저장된 히스토리 재생을 하나의 흐름으로 연결하기
- 데이터가 계속 쌓이는 서비스 특성을 고려해 운영 제한 정책 넣기

## 핵심 기능

### 1. 간단한 입장 흐름

- `index.html -> /paintchat/list -> /paintchat/{roomId}` 구조
- 세션에 `guestNickname`이 없으면 목록/방 접근 시 메인으로 리다이렉트
- 회원가입 없이 즉시 입장 가능한 게스트 기반 UX

### 2. 채팅방 생성과 입장 검증

- 채팅방 생성은 원클릭 방식
- 방 ID는 UUID로 자동 생성
- 화면에는 `0001`, `0002` 형식의 순차 번호 노출
- 입장 시 UUID 형식과 DB 존재 여부를 함께 검증
- 방별로 `이용 가능 여부`를 DB에 저장

### 3. 실시간 그림판

- WebSocket(STOMP) 기반 드로잉 이벤트 정보를 전송
- 사용자 입장 시 브러시 색상 랜덤 지정
- 다른 사용자의 커서 위치를 실시간으로 표시
- 재생 중에는 그림 입력을 막아 상태 충돌을 방지
- 드로잉 샘플링은 `20px`, `100ms` 기준으로 고정

### 4. 그림 이력 저장 및 재생

- 드로잉 데이터를 선분 단위로 DB에 저장
- 방 입장 시 저장된 이력을 순서대로 캔버스에 재생
- 재생이 끝나면 이후 실시간 드로잉을 이어서 반영
- 입장 타이밍이 달라도 같은 그림 맥락을 공유할 수 있도록 설계

### 5. 실시간 접속자 표시

- 방별 접속자 목록을 실시간으로 표시
- 접속/이탈 시 현재 참여자 목록 즉시 갱신
- 다른 사용자의 존재를 텍스트뿐 아니라 커서 움직임으로도 확인 가능

### 6. 운영 제한 정책

- 하루 생성 가능한 채팅방 수 제한: `3개`
- 방당 저장 가능한 드로잉 데이터 제한: `5000개`
- 데이터 한도를 넘기면 해당 방은 자동으로 재생 전용 상태로 전환
- 운영 종료된 방은 그림 감상은 가능하지만 추가 드로잉은 불가

## 구현 포인트

### 1. "늦게 들어와도 이해되는 방" 만들기

실시간 서비스는 보통 "지금 접속한 사람" 중심으로만 동작합니다.  
이 프로젝트에서는 뒤늦게 입장한 사용자도 기존 흐름을 이해할 수 있도록, 드로잉 이벤트를 DB에 저장하고 입장 시 순차 재생하는 구조를 적용했습니다.

이 방식으로 다음 경험을 만들었습니다.

- 이미 그려진 결과를 한 번에 보여주는 것이 아니라 "그려지는 과정"을 보여줌
- 새로 들어온 사용자도 현재 방의 분위기와 흐름을 빠르게 파악 가능
- 재생이 끝난 뒤 자연스럽게 실시간 협업 흐름에 합류 가능

### 2. 실시간성과 저장 비용 사이의 균형

캔버스 입력은 이벤트 수가 매우 많기 때문에, 모든 마우스 이동을 그대로 저장하면 데이터량이 급격히 커집니다.  
이를 완화하기 위해 클라이언트에서 샘플링 간격을 두고 선분을 저장하도록 구성했습니다.

- 최소 이동 거리: `20px`
- 최소 전송 간격: `100ms`

즉, 너무 촘촘한 입력을 그대로 DB에 적재하지 않고, 화면 품질과 저장 비용 사이에서 균형을 잡도록 설계했습니다.

### 3. 서비스 운영 종료 조건을 코드에 반영

그림 채팅은 시간이 지날수록 데이터가 계속 누적되는 구조입니다.  
그래서 방마다 최대 드로잉 데이터 수를 두고, 한계를 넘으면 자동으로 `재생 전용` 상태로 바꾸는 정책을 넣었습니다.

이 정책은 다음 장점이 있습니다.

- DB 적재량이 무한정 늘어나는 것을 방지
- 너무 무거워진 방에서 실시간 드로잉이 계속 발생하는 문제 완화
- 운영이 끝난 방도 콘텐츠처럼 감상 가능한 형태로 유지

## 시스템 구성

### Backend

- `PaintChatPageController`
  - 방 목록 조회
  - 방 생성
  - 방 입장 검증
  - 초기 화면 렌더링
- `PaintChatSocketController`
  - 드로잉 이벤트 수신 및 브로드캐스트
  - 커서 이벤트 브로드캐스트
  - 접속자 입장/이탈 처리
- `PaintChatRoomService`
  - 방 생성
  - 방 번호 증가
  - 일일 생성 제한 관리
- `PaintChatStrokeService`
  - 드로잉 이력 저장
  - 방별 드로잉 개수 제한 관리
- `PaintChatPresenceService`
  - 방별 실시간 접속자 목록 관리

### Frontend

- Thymeleaf 템플릿 기반 화면 구성
- Canvas API로 그림판 구현
- SockJS + STOMP로 실시간 이벤트 송수신
- 입장 시 히스토리 재생, 이후 실시간 드로잉 연결

## 데이터 모델

### `paint_chat_room`

- `roomId`: UUID 기반 고유 식별자
- `roomName`: 화면 노출용 순차 번호
- `enterEnabled`: 입장 가능 여부
- `drawEnabled`: 드로잉 가능 여부
- `createdAt`: 생성 시각

### `paint_chat_stroke`

- `roomId`: 소속 방
- `guestNickname`: 그린 사용자 닉네임
- `color`, `size`: 브러시 정보
- `fromRatioX`, `fromRatioY`, `toRatioX`, `toRatioY`
  - 화면 크기가 달라도 복원할 수 있도록 좌표를 비율로 저장
- `createdAt`: 저장 시각

## 화면 흐름

### 메인 페이지

- 게스트 닉네임 표시
- 그림 채팅방 목록으로 이동

### 채팅방 목록 페이지

- 원클릭 방 생성
- 생성된 방 목록 조회
- 방 운영 상태 확인
- 메인 페이지 복귀

### 채팅방 페이지

- 저장된 그림 순차 재생
- 실시간 공동 드로잉
- 실시간 커서 위치 표시
- 실시간 접속자 목록 표시
- 재생 중 드로잉 차단

## 프로젝트의 주요 포인트

- 실시간 WebSocket 기능과 DB 영속화를 함께 다룬 프로젝트
- 단순 CRUD가 아니라 "상태 변화"와 "동시성 흐름"을 고려한 서비스
- 사용자 경험을 위해 그림 결과가 아닌 "그려지는 과정"을 복원하도록 설계
- 운영 정책을 데이터 모델과 서비스 로직에 직접 연결
- 서버 렌더링 구조 안에서도 실시간 상호작용 구현

## 이후 확장 방향

현재는 선분 단위 저장 방식이기 때문에 장시간 운영 시 데이터량이 빠르게 늘어날 수 있습니다.  
이후에는 아래 방향으로 개선할 수 있습니다.

- 일정 시점마다 캔버스 스냅샷 저장
- 오래된 선분을 스냅샷으로 합쳐 재생 부담 줄이기
- 관리자용 방 제어 기능 추가
- 방별 옵션 설정 기능 추가
- 드로잉 데이터 압축 또는 스트로크 단위 저장 구조 도입
